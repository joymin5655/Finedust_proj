database 1. Data Sources & Collection

AirLens는 4개의 공개 데이터 소스에서 대기질·기상·위성 데이터를 자동 수집합니다. 모든 API 키는 GitHub Secrets에만 저장되며, 클라이언트 코드에 노출되지 않습니다.

sync 2. Data Pipeline

GitHub Actions가 외부 API에서 데이터를 수집하고, 정규화된 JSON으로 변환하여 저장합니다. 브라우저에서는 FusionService가 이 데이터를 하나의 통합 뷰로 합성합니다. 

External APIs                    Automated Pipeline          Cached Data             Browser
─────────────                    ──────────────              ───────────             ───────
WAQI API ──────┐
OpenAQ API ────┤                 GitHub Actions              Static JSON
Earthdata API ─┤  ──→  Fetch & Normalize  ──→  Store  ──→  FusionService
Open-Meteo ────┘       (daily/weekly)         (per source)   ↓
                                                             Multi-source
                                                             merge + DQSS
                                                                  ↓
                                                             Globe / Today
                                                             Policy / Charts
① 수집 (Ingestion)

GitHub Actions 워크플로우가 스케줄에 따라 외부 API를 호출하고 원본 응답을 JSON으로 저장합니다. Rate limit을 준수하며, 실패 시 자동 재시도합니다.

② 정규화 (Harmonization)

Python/Node 스크립트가 다양한 API 형식을 통일된 스키마로 변환합니다. 좌표 정규화(WGS84), 단위 변환(µg/m³), 중복 제거를 수행합니다.

③ 융합 (Fusion)

브라우저의 FusionService가 WAQI + OpenAQ + Open-Meteo 데이터를 좌표 기반으로 병합하고, 각 관측소에 데이터 품질 점수(DQSS)를 자동 부여합니다.

analytics 3. Analysis Engines

AirLens는 6개의 분석 엔진을 통해 데이터 품질 평가, 예측, 정책 효과 분석을 수행합니다.

1

Policy Impact Analysis

✅ 66 COUNTRIES

왜 필요한가: 정부가 대기질 개선 정책을 시행해도, 실제 효과를 수치로 검증하기 어렵습니다. 자연 변동, 기상 조건, 경제 변화 등 많은 요인이 얽혀 있어 단순 전후 비교로는 부족합니다.

AirLens는 66개국의 환경 정책 시행 전후 대기질 변화를 통계적으로 비교하여, 정책이 실질적 효과가 있었는지 데이터 기반으로 평가합니다. 이중차분법(DID) 기반 분석으로 정책 외 다른 요인의 영향을 분리합니다.

⚠️ 정책 분석 결과는 준실험적(approximate causal) 추정치입니다. 단독 정책 근거로 사용하지 마세요.

2

Satellite → PM2.5 Estimation

🔬 DEVELOPING

왜 필요한가: 전 세계 인구의 90% 이상이 대기질 측정소가 없는 지역에 살고 있습니다. 위성 데이터(AOD)는 전 지구를 커버하지만, 이를 지상 PM2.5 농도로 변환하려면 기상 조건과 지형 보정이 필수입니다.

AirLens는 위성 에어로졸 데이터에 기상 변수(온도, 습도, 풍속, 대기경계층 높이)와 지리 정보를 결합하여 센서 없는 지역의 PM2.5를 추정합니다. 모든 추정값에는 불확실성 범위가 함께 표시됩니다.

3

Camera AI — Image-based AQ Classification

⚗️ EXPERIMENTAL

왜 필요한가: 하늘 사진 한 장으로 대기질을 즉시 판단할 수 있다면, 누구나 스마트폰 하나로 자신의 대기 환경을 확인할 수 있습니다. 대기 오염 물질은 빛의 산란, 가시거리, 하늘 색상에 직접적인 영향을 미칩니다.

딥러닝 모델이 하늘 이미지에서 연무, 가시거리, 색상 패턴을 분석하여 대기질 등급을 분류합니다. 모든 추론은 브라우저 내에서 실행되며, 사진이 서버로 전송되지 않습니다.

4

DQSS — Data Quality Scoring System

✅ ACTIVE

왜 필요한가: 모든 센서 데이터가 동일하게 신뢰할 수 있는 것은 아닙니다. 오래된 데이터, 결측이 많은 관측소, 다른 소스와 불일치하는 값은 그대로 사용하면 잘못된 결론을 낳습니다.

DQSS는 5가지 기준(신선도, 완전성, 일관성, 안정성, 모델 잔차)으로 각 관측소의 데이터를 0–100점으로 평가합니다. 높은 품질의 데이터에 더 큰 가중치를 부여하여 분석 신뢰도를 높입니다.

5

Bayesian Reliability Engine

📋 PLANNED

왜 필요한가: 센서의 신뢰도는 시간에 따라 변합니다. 새 센서일 때는 정확하지만, 점차 드리프트가 발생할 수 있습니다. 관측 데이터가 쌓일수록 각 센서의 신뢰도를 학습하여 더 정확한 품질 평가를 제공할 예정입니다.

6

Anomaly Detection

📋 PLANNED

왜 필요한가: 센서 오작동, 갑작스런 환경 이벤트(산불, 황사), 데이터 전송 오류를 자동으로 식별해야 합니다. 이상치가 제거되지 않으면 평균값과 분석 결과를 왜곡시킵니다. 향후 시계열 패턴 분석과 다변량 이상치 탐지를 결합하여 자동 품질 관리를 수행할 예정입니다.
